# 首页个股删除二次确认

## Goal

降低首页个股栏误删除历史记录的风险，让用户在真正执行不可逆删除前明确核对并确认删除对象。

## Background

- 用户反馈首页个股栏目点击删除后会直接执行，并且经常发生误删除。
- 首页 `StockBar` 有两类删除入口：卡片内单条删除，以及勾选一个或多个条目后的批量删除（`apps/dsa-web/src/components/history/StockBar.tsx`、`StockBarItem.tsx`）。
- 首页还会把“大盘复盘（MARKET）”合并进同一栏目；上述入口最终通过 `HomePage.handleDeleteStock` 调用 `historyApi.deleteByCode(stockCode)` 删除该代码的全部历史记录（`apps/dsa-web/src/pages/HomePage.tsx:414`）。
- 项目已有通用 `ConfirmDialog`，并支持危险操作样式和禁用确认/取消按钮，可复用现有交互与视觉规范（`apps/dsa-web/src/components/common/ConfirmDialog.tsx`）。

## Requirements

- R1：卡片内单条删除、勾选后的批量删除，以及“大盘复盘（MARKET）”删除，都必须在用户明确确认后才调用删除接口。
- R2：所有删除始终要求二次确认，不提供“本次不再提示”或任何持久化绕过设置。
- R3：确认弹窗必须明确提示删除后不可恢复，并展示足以核对目标的信息：
  - 单条删除显示目标名称和代码。
  - 批量删除显示目标总数和前 5 个名称/代码；超过 5 个时显示剩余数量。
- R4：确认按钮使用红色危险样式，中文文案为“确认删除”；取消按钮保持普通样式。
- R5：取消确认或点击遮罩关闭弹窗时，不得调用删除接口，不得改变当前列表和勾选状态。
- R6：确认删除后只删除弹窗列出的目标；删除执行期间必须避免重复提交。
- R7：删除成功后保持现有刷新行为，包括个股栏、历史记录，以及删除 `MARKET` 后的大盘复盘历史刷新。
- R8：新增确认文案必须同时提供中文和英文版本，并沿用现有 UI 文案机制。

## Acceptance Criteria

- [x] AC1（R1、R3）：点击任一单条删除入口后先显示目标名称和代码；确认前 `historyApi.deleteByCode` 不被调用。
- [x] AC2（R1、R3）：点击批量删除后显示选中总数、前 5 项预览和必要的剩余数量；确认前不调用删除接口。
- [x] AC3（R1）：删除“大盘复盘（MARKET）”时同样先显示确认弹窗。
- [x] AC4（R2）：界面不存在跳过确认或记住确认选择的入口。
- [x] AC5（R4）：确认按钮显示为危险操作样式，中文文案为“确认删除”。
- [x] AC6（R5）：点击取消或遮罩关闭弹窗后不删除记录，批量勾选状态保持不变。
- [x] AC7（R6、R7）：点击确认后只删除弹窗所列目标，并沿用现有刷新逻辑；删除进行中不能重复确认或取消。
- [x] AC8（R8）：中英文界面均能展示完整、可理解的删除确认标题、说明和按钮文案。
- [x] AC9：前端定向测试覆盖单条打开与确认、批量预览与确认、取消不删除、`MARKET` 删除确认，以及执行中禁用状态。

## Out of Scope

- 不改变后端删除 API、数据存储或删除范围语义。
- 不增加回收站、撤销删除或历史恢复能力。
- 不调整首页个股栏之外的删除交互。
- 不增加绕过二次确认的用户设置。

## Technical Notes

- 这是局部 Web 交互优化，按轻量任务处理，规划阶段仅维护 PRD。
- 批量勾选状态当前由 `StockBar` 内部维护；确认交互需要保证取消后仍保留这些勾选。
- `ConfirmDialog` 的 `isDanger`、`confirmDisabled` 和 `cancelDisabled` 能覆盖危险样式与执行中防重复要求。
