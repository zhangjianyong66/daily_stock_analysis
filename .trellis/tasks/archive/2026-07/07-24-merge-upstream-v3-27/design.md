# 技术设计

## 1. 集成边界

- 当前基线：本地 `main` / `origin/main` 的 `0fa4a41`。
- 目标基线：上游正式 `v3.27.0` 解引用提交 `b36c721`。
- 共同祖先：`bfdee032`。
- 集成方式：从当前 `main` 创建 `integrate/upstream-v3.27`，以一次 `git merge --no-ff` 引入目标提交；不改写已有本地提交。
- 交付边界：合并提交和本地验证只落在集成分支，等待用户审阅后再决定是否推进 `main` 或推送。

## 2. 保留与替换原则

本地有效能力默认保留。合并时按业务契约判断，而不是按文件归属或提交时间判断：

- ETF 资金流、实时行情降级、搜索审计与污染隔离、Vision 异步截图导入、Docker 代理/端口适配、首页移动端交互和排序置顶等本地能力继续存在。
- 上游的市场结构、决策信号、Multi-Agent、Codex App Server、Futu 持仓、首页工作区等新增能力接入现有流程。
- 同一契约出现两套实现时，优先保留向后兼容的入口和字段；只有本地实现已废弃，或上游实现已完整覆盖并通过回归，才删除本地实现。
- 不引入 `b36c721` 之后的上游 `main` 提交。

## 3. 已知冲突的语义合并

| 文件 | 本地能力 | 上游能力 | 处理方式 |
| --- | --- | --- | --- |
| `apps/dsa-web/src/i18n/uiText.ts` | 批量分析、移动端、搜索审计、删除确认、排序和置顶文案 | 决策风格、首页工作区、Codex 文案 | 按 key 合并，保持 key 唯一；保留两侧文案和语言类型契约 |
| `apps/dsa-web/src/pages/HomePage.tsx` | 移动端布局、筛选、排序、置顶、批量分析 | 历史/自选/今日工作区、批量任务状态 | 以状态流和请求生命周期为主线重组，避免重复请求、旧响应覆盖新状态和布局回退 |
| `src/services/system_config_service.py` | Vision 中转站 Responses 配置、Extra Headers 等边界 | Codex/MiniMax 配置状态和能力检测 | 保留独立配置解析与能力检测入口，统一返回兼容字段，不用域名或模型名猜测协议 |

## 4. 无文本冲突的语义审查

36 个双方共同修改文件中，除上述 3 个直接冲突外，重点审查：

- Agent 编排、工具面、决策信号、报告和历史载荷之间的字段兼容。
- `config.py`、系统配置 API、`.env.example`、Docker 和工作流之间的配置映射。
- 搜索服务、数据源 fallback、报告渲染、通知模板是否仍遵守本地稳定性护栏。
- Web API 类型、页面状态、桌面端入口和测试是否同步。

自动合并成功不等于语义合并成功；每个涉及上述区域的文件都要有 diff 审查记录或对应回归证据。

## 5. 兼容性与数据边界

- API 优先采用追加字段和兼容解析，保留现有客户端可消费的字段及枚举。
- 报告和 DecisionSignal 继续保持现有历史数据读取兼容；新上游字段缺失时走既有默认或诊断路径。
- 不执行数据库清理、历史数据迁移或生产部署；如上游代码包含迁移逻辑，仅在测试环境确认可重复执行和回滚边界。
- 不改变 `.env` 真实密钥、端口和本地部署状态。

## 6. 回滚设计

- 合并前在本地创建指向 `0fa4a41` 的备份分支，保留集成分支和备份分支的可追溯关系。
- 合并尚未提交时使用 `git merge --abort`，恢复集成分支到合并前状态。
- 合并提交后、尚未推进 `main` 时，直接切回 `main`，集成分支保留供修复；必要时在集成分支上修订，不重置共享分支。
- 若未来用户批准将合并提交带入 `main` 后发现问题，优先通过带主线父节点的 revert 回滚，并保留原合并提交作为审计证据。
