# 修复分析任务长期执行中

## Goal

修复 Web/API 分析任务可能长期停留在“执行中”的问题，同时治理底层数据源调用的超时、重试与 fallback 语义，让任务队列在底层分析卡住、第三方数据源阻塞、线程池异常或进程重启后都能给出可解释、可恢复的终态或明确的超时状态。

用户价值：

- 用户不再看到已经无实际进展的任务无限显示“执行中”。
- 重复提交同一股票时，不会因为旧任务残留而长期被“正在分析中”拦截。
- 运维排查时能从任务状态、运行流或日志中看出任务是超时、失败、过期还是已经完成。

## Requirements

- R1：普通分析任务不得无限期停留在 `pending` 或 `processing`。超过可配置或明确默认的最大运行时长后，任务必须进入可解释的终态或超时状态。
- R2：任务超时、失败或取消后，必须释放 `_analyzing_stocks` 去重占用，允许用户重新提交同一股票。
- R3：任务状态、运行流和日志必须包含足够信息区分“仍在执行”“已超时失败”“任务已过期/不存在”“历史记录已完成”。
- R4：修复必须覆盖普通股票分析入口，不能只修复本次 `159865` 个例。
- R5：默认行为应保持现有成功任务路径兼容，不改变已完成任务落库和历史查询语义。
- R6：新增配置项时，必须同步更新 `.env.example` 和相关文档；若不新增配置项，应在设计中说明默认超时来源。
- R7：修复应包含回归测试，至少证明长时间卡住的任务会被标记为失败/超时，并释放重复提交锁。
- R8：超时后的用户可见状态复用现有 `failed`，不新增 `timeout` 枚举；超时原因通过 `error` / `message` 明确表达。
- R9：底层数据源调用必须按能力维度有明确超时预算，至少覆盖本次事故证据链涉及的股票名称获取，以及普通分析主路径中的日线数据、实时行情。
- R10：数据源 timeout 的处理策略必须避免堆叠悬挂调用。已有 provider 内部请求级 retry 保留；manager 层遇到调用超时后，应记录失败、短期降级该 provider，并切换下一个可用 provider，而不是立即对同一 provider 再开一次可能继续悬挂的调用。
- R11：新增超时配置必须有安全默认值，且同步更新 `.env.example`、相关配置解析与 `docs/CHANGELOG.md`。
- R12：迟到完成的底层线程不得覆盖已经超时失败的任务状态，也不得释放同一股票后续新任务的 `_analyzing_stocks` 占用。

## Acceptance Criteria

- [ ] 给定一个不会返回的分析 callable 或模拟阻塞任务，超过任务最大运行时长后，任务状态不再是 `pending` / `processing`。
- [ ] 超时后的任务包含明确错误信息，用户可从任务状态或运行流看到超时原因。
- [ ] 超时后，同一 `stock_code` 可以重新提交，不会被旧任务的 `_analyzing_stocks` 占用拦截。
- [ ] 正常完成的任务仍保持 `completed`、`progress=100`、结果可读、历史运行流兼容。
- [ ] 任务列表和运行流接口对超时任务返回一致状态，不出现前端一直显示“执行中”的状态漂移。
- [ ] 后端测试覆盖任务超时与重复提交释放；现有任务队列相关测试仍通过。
- [ ] 底层数据源 manager 级调用超时后，会继续尝试下一个 provider；对应 provider 失败会被记录到运行流或日志中。
- [ ] manager 级超时不会让同一个 provider 的立即重复调用无限堆叠；至少在当前请求链路内跳过/熔断该 provider 并继续 fallback。
- [ ] 迟到返回的旧任务不会把已超时任务改回 completed，也不会删除新任务的去重锁。
- [ ] 新配置项在 `.env.example` 中有说明，并在 changelog 的 `[Unreleased]` 段记录。

## Out of Scope

- 不在本任务中重构全部数据源 fallback 架构；本次只补齐普通分析主路径的超时与 fallback 护栏。
- 不在本任务中实现跨进程/跨重启的完整持久化任务队列。
- 不在本任务中修改股票分析报告内容或投资结论生成逻辑。
