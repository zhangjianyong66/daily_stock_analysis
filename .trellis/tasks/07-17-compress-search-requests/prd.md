# 压缩大盘复盘与分析搜索请求

## Goal

在不降低大盘复盘和个股分析所需新闻情报覆盖、不破坏搜索供应商降级语义的前提下，减少真实外部搜索请求及对应费用，并让优化前后的物理请求数量可以通过 `search_api_calls` 审计验证。

## Background

- 2026-07-17 15:00 的 A 股大盘上下文生成只启动了一次，但 `MarketAnalyzer.search_market_news()` 依次执行了三个主题查询，共产生三次成功的 Anspire 物理请求。
- 三个现有主题分别为 `A股 大盘 复盘`、`股市 行情 分析`、`A股 市场 热点 板块`；每次请求 `top_k=6`，三次共最多获取 18 条原始候选，最终准入数量分别为 1、2、3 条。
- 三条审计记录均为首次物理请求成功，不是 retry 或 provider fallback；页面因只显示相同的来源和操作而看起来像重复调用。
- 15:00–16:00 的普通分析审计包含 12 只 ETF，每只冷启动按 `fresh_events`（近 3 天）和 `analysis`（近 30 天）两组执行，共 24 次 Anspire 物理请求；这两个时间窗口不是 retry 或 provider fallback。
- 2026-07-17 当天有 3 只 ETF 在首次分析后的 4–11 分钟内再次产生完整的两组请求，共 6 次可疑重复物理调用。现有可信结果缓存属于单个 `SearchService` 实例，无法在同一进程内的不同分析任务实例之间复用。
- 非 ETF 的标准 Pipeline 在 `max_searches=5` 时仍按五个逻辑维度逐次请求；本任务在 Anspire 可用时按近期事件与中期分析分组，但必须保留公司身份、最新消息、公告、风险、机构分析和业绩情报契约。

## Requirements

- R1：每个市场的一次大盘复盘只执行一次综合新闻搜索，综合查询必须覆盖原有行情复盘、市场分析和热点板块三个意图。
- R2：单次综合搜索应保留足够候选数量，并继续经过现有时效、语言、相关性、质量和结果数量过滤。
- R3：分析任务请求优化必须先基于真实 `search_api_calls` 审计数据形成按入口和股票类型拆分的请求基线，再选择可合并点。
- R3.1：ETF 必须保持近 3 天事件与近 30 天分析两个独立时间窗口；优先通过进程级可信结果缓存和同键并发合并消除重复任务请求，而不是把两个窗口强制压成一个。
- R3.2：标准 Pipeline（`call_source=analysis`）的非 ETF Anspire 五维搜索可按相同时效语义合并物理请求，但合并后的本地分类必须继续产出既有逻辑维度，且未知日期和严格时效规则不得放宽。Agent 六维工具链本次保持现状。
- R4：不得将 provider retry、备用 Key、供应商 fallback、不同时间窗口或缓存 miss 等不同物理语义静默合并。
- R5：优化后继续遵守“一次真实外部 HTTP 请求一条审计记录”的计数契约；缓存命中不得伪造物理请求记录。
- R6：用户可见的搜索行为或报告情报来源发生变化时，同步更新相关文档与 `docs/CHANGELOG.md`。

## Acceptance Criteria

- [x] AC1：大盘复盘单市场新闻搜索的物理请求基线从 3 次降为 1 次，并有单元测试锁定调用次数、综合查询意图和 `call_source=market_review`。
- [x] AC2：大盘综合搜索使用单次逻辑上限 6、Provider `top_k=12`，与 2026-07-17 三次查询合计准入 6 条的实际基线一致；相关单元测试锁定候选和最终上限。
- [x] AC3：输出普通分析任务当前请求基线，至少区分普通股票、ETF、Agent/工具调用、cache、retry 和 provider fallback。
- [x] AC4：同一进程内，不同 `SearchService` 实例对同一 ETF、同一 profile、同一分组和同一时间窗口的有效缓存可以复用；并发冷启动只允许一个 owner 发出物理请求，失败或空结果不得缓存。
- [x] AC5：ETF 单只冷启动仍最多两次物理请求，15 分钟内重复分析在两组均命中时为 0 次；近 3 天组过期而近 30 天组有效时只刷新 1 次。
- [x] AC6：标准 Pipeline 的非 ETF 分析在 Anspire 两组请求成功时，单只冷启动由 5 次降为 2 次，并按市场保留既有五维：A 股为最新消息、机构分析、风险、公告、业绩，海外为最新消息、机构分析、风险、业绩、行业；Anspire 物理失败时只对失败组保留既有 Provider 降级并如实审计，非 Anspire provider 与 Agent 路径行为不变。
- [x] AC7：对普通分析中无法安全合并的请求在设计中记录原因，不以隐藏审计记录或跳过必要搜索实现降本。
- [x] AC8：相关离线测试和后端完整门禁通过；若执行在线 Anspire smoke，必须仅报告脱敏后的请求次数和覆盖结论。
- [x] AC9：`docs/CHANGELOG.md` 与受影响的搜索行为文档已同步，回滚方式明确。

## Out Of Scope

- 修改 Anspire 计费规则、供应商账户或 API Key。
- 通过关闭搜索功能、缩短既有新闻时效窗口或删除必要情报维度来达到请求下降。
- 将网络失败后的必要重试和跨供应商 fallback 从审计中隐藏。
- 把 ETF 近 3 天与近 30 天双窗口强制合并为一次。
- 修改 Agent 六维工具链的请求分组、维度顺序或 `industry` 时间语义。
